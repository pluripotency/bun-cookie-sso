app1.test.localとapp2.test.localをローカルで試すためにはnginx等のreverse proxyがないとできないでしょうか。


結論から言うと、**Nginxなどのリバースプロキシがなくても、ブラウザとNode.js（Bun）の設定だけでテスト可能**です。

ブラウザは「アクセスしているドメイン名」を見てCookieを送るかどうかを判断するため、リバースプロキシの代わりに**ホストファイル（hosts）**を書き換えるだけで、ローカル環境でも擬似的にサブドメイン環境を作れます。

---

### ステップ1：ホストファイル（hosts）の編集

PCのOSに「`app1.test.local` は自分のPC（127.0.0.1）だよ」と教えます。

* **Windows**: `C:\Windows\System32\drivers\etc\hosts` を管理者権限で開く
* **Mac/Linux**: `/etc/hosts` を `sudo` で開く

末尾に以下の2行を追記して保存してください。

```text
127.0.0.1 app1.test.local
127.0.0.1 app2.test.local

```

### ステップ2：コードの微調整

前のコードの `DOMAIN` 指定を、テスト用に作成したドメインに変更します。

```typescript
const DOMAIN = '.test.local'; // ドットから始めることでサブドメイン間で共有可能

```

### ステップ3：動作確認のテスト

1. **Bunを起動**: `bun index.ts`
2. **ログイン**: ブラウザで `http://app1.test.local:3000/login` にアクセス。
3. **Cookieの確認**: ブラウザのデベロッパーツール（F12）の「アプリケーション」→「Cookie」を開き、Domainが `.test.local` になっていることを確認。
4. **別サイトでの確認**: ブラウザの別タブで `http://app2.test.local:3000/check-auth` にアクセス。

リバースプロキシがなくても、これで「app2でも認証が通っている」状態を再現できます。

---

### なぜプロキシがいらないのか？

ブラウザのCookie共有ロジックは以下の通りです：

1. ブラウザが `app1.test.local` にリクエストを送る。
2. サーバーが `Domain=.test.local` という指示を出す。
3. ブラウザは「次に `*.test.local` に行くときはこのCookieを添えよう」と保存する。
4. ブラウザで `app2.test.local` を開くと、ブラウザ自身が「あ、これは `.test.local` の一部だ」と判断してCookieを自動で送信する。

### 注意点：ポート番号について

通常、Cookieはポート番号を無視してドメインのみで判断されます。そのため、同じポート（3000）で動いていても、ドメイン名さえ違えば別サイトとしての挙動をテストできます。

もし「ポート3000を見せたくない（`http://app1.test.local` だけでアクセスしたい）」という場合は、Nginxを挟んで80番ポートで受ける設定が必要になります。
